# 중앙 RAG 서버 아키텍처 전환

**날짜**: 2026-04-01
**작업 개요**: context_search.exe에 HTTP 서버 모드(--serve)와 클라이언트 모드를 추가하여, 공용 서버 1대에서 RAG를 중앙 관리하고 팀원 PC는 HTTP로 검색하는 아키텍처로 전환.

## 수정 파일

### `mcp/context_search/server.py` (핵심 변경)
- **3모드 지원**: 로컬 / 서버(--serve) / 클라이언트(HTTP 위임)
- **DoubleBufferedIndex 클래스**: Live/Work 이중 컬렉션으로 무중단 인덱싱
  - 하나의 ChromaDB에 `context_a` / `context_b` 두 컬렉션 사용
  - 검색은 항상 Live에서 즉시 응답 (블로킹 없음)
  - 갱신은 Work에서 진행 → 완료 시 원자적 교체
  - 태그 캐시도 인메모리에서 동시 교체
  - 기존 `context` 단일 컬렉션에서 자동 마이그레이션
- **FastAPI HTTP 서버**: 8개 엔드포인트 (검색 4, 인덱싱 2, 상태 2)
- **HTTP 클라이언트**: urllib.request 기반, 각 MCP 도구에 원격 분기 추가
- **config.json / 환경변수에서 서버 URL 자동 탐색**

### `watcher/watch.py`
- `server_mode` 설정 시 `context_search.exe --serve`를 백그라운드 프로세스로 시작
- `update_vector_index()` / `search_related_contexts()` 에 HTTP 호출 경로 추가
- `load_or_init_config()`에 서버 모드 설정 프롬프트 추가
- 서버 프로세스 시작 대기 (health check) 및 종료 정리

### `build.bat`
- `pip install` 라인에 `fastapi uvicorn` 추가
- `context_search.exe` 빌드에 `--hidden-import=fastapi,uvicorn` 추가

### `CLAUDE.md`
- 서버/클라이언트 모드 설명 추가
- config.json 스키마 갱신 (server_mode, server_host, server_port, context_server_url)
- HTTP 엔드포인트 테이블, 더블 버퍼링 설명 추가
- 핵심 설계 결정 테이블에 4개 항목 추가

## 주요 설계 결정

| 결정 | 이유 |
|------|------|
| context_search.exe 1개에 3모드 통합 | 배포 단순화 — 하나의 dist/로 서버/클라이언트/로컬 모두 지원 |
| 컬렉션 레벨 이중화 (context_a/b) | 디렉토리 복사 대신 → Windows 파일 잠금 문제 회피 |
| 임베딩 포함 복사 (get + add with embeddings) | 재임베딩 불필요, O(n) 데이터 복사만 |
| urllib.request (stdlib) | 클라이언트에 추가 의존성 없음 |
| FastAPI + uvicorn | 비동기 지원, 자동 OpenAPI 문서, 경량 |
| watch.py가 HTTP 서버를 subprocess로 시작 | watch.exe 빌드 크기 유지, 관심사 분리 |
