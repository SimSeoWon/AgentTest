# stem 단위 컨텍스트 MD 그룹핑으로 전환

## 작업 개요

컨텍스트 MD 생성 시 디렉토리 단위 그룹핑에서 stem(파일명) 단위 그룹핑으로 변경.
기존 방식에서는 한 디렉토리의 일부 파일만 변경될 때, 변경되지 않은 파일의 컨텍스트가 덮어씌워져 유실되는 문제가 있었음.

## 문제 상황

- 커밋 A: `Combat/DamageSystem.cpp` + `Combat/HitDetection.cpp` 변경 → `Combat.md` (두 파일 분석)
- 커밋 B: `Combat/DamageSystem.cpp`만 변경 → `Combat.md` 덮어쓰기 (HitDetection 분석 유실)
- RAG 네비게이션에서 HitDetection 관련 검색 결과가 사라짐

## 해결 방법

`_group_files`를 항상 stem 단위로 그룹핑하도록 변경:
- `DamageSystem.h` + `DamageSystem.cpp` → `Combat/DamageSystem.md`
- `HitDetection.h` + `HitDetection.cpp` → `Combat/HitDetection.md`
- 각 파일이 독립적인 MD를 가지므로 다른 파일 변경에 영향받지 않음

## LLM 교차 폴백 추가

`_call_llm`에서 primary LLM 실패 시 다른 LLM으로 자동 재시도:
- Claude 실패 → Gemini 폴백, Gemini 실패 → Claude 폴백
- `_call_gemini`에서 Gemini CLI 미설치 시 바로 Claude 호출하던 로직 제거 (폴백 책임을 `_call_llm`으로 일원화)
- 양쪽 모두 실패 시 `None` 반환 → 해당 그룹 건너뜀

## 수정 파일

- `watcher/context.py` — `_group_files` 함수 단순화, `initial_context_build` 변수명 정리, `update_vector_index` 주석 갱신
- `watcher/common.py` — `_call_llm` 교차 폴백 추가, `_call_gemini` 자체 폴백 제거
- `CLAUDE.md` — 그룹핑 방식 및 LLM 폴백 설명 갱신

## 설계 결정

- 토큰 비용: stem 단위가 호출 수는 많지만, 변경 안 된 파일을 재분석하지 않아 총 토큰은 동등 또는 감소
- 하위 호환: `update_vector_index`에서 기존 디렉토리 단위 MD 탐색 로직 유지 (이전 버전 MD도 인덱싱)
- `GROUP_CONTENT_LIMIT` 상수는 더 이상 그룹핑 분할 기준으로 사용되지 않으나, LLM 프롬프트 크기 제한(`code_block`)에서 여전히 사용됨
