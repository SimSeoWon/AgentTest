# 2026-03-29 12:00 — 리뷰 상담 에이전트 및 코멘트 시스템 추가

## 작업 개요
코드 리뷰 지적 사항 대응뿐 아니라, 개발자의 설계 의도·고민·방향성·진행 상황을
컨텍스트 MD 파일에 기록하는 코멘트 시스템을 구현.
코멘트 유형 태그(`[리뷰]`, `[방향]`, `[고민]`, `[진행]`, `[메모]`)로 분류하여
다음 리뷰 및 다른 에이전트가 코드의 맥락 지식으로 활용하도록 함.

## 변경 전후

| 항목 | 변경 전 | 변경 후 |
|------|--------|--------|
| 에이전트 수 | 10개 (01~10) | 11개 (01~11, 11_리뷰상담 추가) |
| 컨텍스트 MD 구조 | 요약 + 개선필요사항 | + `## 코멘트` 섹션 (선택) |
| 벡터 임베딩 | 본문 전체 임베딩 | `## 코멘트` 섹션 제외 후 임베딩 |
| 코드 리뷰 프롬프트 | 컨텍스트만 참조 | 컨텍스트 + 기존 코멘트 참조 |
| 컨텍스트 갱신 | MD 전체 덮어쓰기 | 기존 코멘트 보존 후 재부착 |

| `CLAUDE.md` | 11_리뷰상담 에이전트 문서화, 컨텍스트 MD 스키마에 코멘트 섹션 추가 |

## 주요 설계 결정

| 결정 | 이유 |
|------|------|
| 코멘트를 컨텍스트 MD에 직접 기록 | 별도 파일 관리 불필요, 소스 파일과 1:1 대응 유지 |
| 벡터 임베딩에서 코멘트 제외 | 주관적 의견이 의미 검색 품질을 저하시키지 않도록 |
| 태그 검색에는 코멘트 포함 | 에이전트가 파일 조회 시 코멘트를 참조할 수 있도록 |
| `update_context()` 시 코멘트 보존 | 소스 재분석으로 개발자 의견이 소실되지 않도록 |
| 리뷰 프롬프트에 코멘트 주입 | "인지된 항목" 반복 지적 방지 + 설계 맥락 이해 |
| 01_소스분석에 코멘트 미생성 지시 | LLM이 임의로 코멘트 섹션을 만들지 않도록 방어 |
| 코멘트 유형 태그 5종 도입 | 리뷰 응답 외에 설계 방향·고민·진행·메모를 구조적으로 기록 |
